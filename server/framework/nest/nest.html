1. 基本概念 ​

1.1. MVC 三层架构 ​

MVC 是一种架构的思想，不是具体的实现。

  • M：Modal，数据/业务/逻辑，负责处理数据逻辑或业务逻辑的工作，其中整个 Modal 又拆分为业务 service 处理层和模型 DAO 层
  • V：View，视图，前后端未分离前，代表着 Jsp 视图；分离后由前端进行处理渲染
  • C：Controller，控制器，客户端的 HTTP 请求最先经过控制器，控制器负责分发调度处理请求

一次完成的请求流程：

  1. 客户端发送 HTTP 请求
  2. 控制器接收到请求，根据请求的内容不同，调用 Modal 处理业务
  3. 业务 service 层，进行业务逻辑的处理，调用模型 DAO 层，对数据库进行操作
  4. 模型 DAO 层处理完数据库之后，把结果依次响应返回 DAO -> service -> controller
  5. 此时，控制器还未能响应结果，需要根据模型响应的数据判断是否需要响应视图。需要展示视图，控制器调度视图组件（比如Jsp）展示并反馈回 controller；不需要展示视图，控制器直接将数据经过响应体返回给客户端。

1.2. 名词解释 ​

名称简要概括用途
PO持久化对象数据库中的表映射的对象类，表的字段与 PO 对象的属性一一对应
VO值对象用于业务层之间的数据传递，比如响应给客户端的数据对象，主要对应界面的显式数据
DTO数据传输对象经过处理后（增加/减少属性）的 POJO 对象，用于跨进程或网络传输的对象载体，比如前端传递的请求体参数，用 DTO 承载。DTO、VO、POJO 之间的关系可以进行转换，比如展示在视图层时，POJO 的身份就是 VO；承载请求响应体参数、跨进程参数传递时，POJO 的身份就是 DTO
DAO数据访问对象为业务层提供接口，负责和底层数据库进行交互，封装简单的 CRUD 方法
BO业务对象由 service 层输出的封装业务的对象，这个对象可以包括或多个其他对象，比如一个学生对象有班级、课程表等信息，那么班级就是一个 PO，课程表是一个 PO，建立一个学生对应的 BO 对象就是处理这些封装的信息
POJO简单的对象可以转化为 PO、DTO、VO，比如 POJO 在传输时就是 DTO，响应回给视图层展示时就是 VO

分层之间和名词的对应关系：

  • 业务层：【拆分 service 和数据模型】，POJO、PO、DTO、BO
  • 视图层：前端负责，包含 VO、DTO
  • 控制层：包含 DTO、POJO、VO

在上面花了大量的篇幅介绍了 MVC三层架构 的概念，那为什么要这么设计呢？

首先明确一点，一个应用发展到庞大的体量时，如果不进行解耦分层设计，那么带来的维护难度将是不可估量的，耦合度也很高。为了解决这个痛点，MVC架构 出现了，而且普遍应用于市场上的很多互联网应用程序。

INFO

一个架构或技术的出现，必定伴随着它的利与弊，MVC架构 所带来的降低项目复杂度、解耦合的优点，是建立在架构复杂的基础上，需要花费一定时间去思考设计应用架构，不利于中小型软件的快速开发，调试起来也增加了复杂度，但这不影响它是最常用的架构设计。

在 nest 应用中，MVC架构 是怎么体现的呢？

可以简单理解为：

  • *.controller.ts：对标控制 controller 层，用户访问时最先经过控制器，由控制器调度分发给其他任务处理，简单对标为前端项目的路由。
  • *.service.ts：对标服务 service 层，封装对数据库的 CRUD 的方法，与 *.entity.ts 属于同一个层。
  • *.entity.ts：对标持久化对象 PO ，映射数据库表，表的字段与该对象的属性一一对应。
  • *.dto.ts：对标数据传输对象 DTO。
  • *.module.ts：模块，对应用程序的拆分为每个小的模块，这些模块会成为 nest 应用的基座，简单对标前端的页面、功能模块。

2. nest 设计模式 ​

nest 应用中默认使用的设计模式——IoC控制反转和DI依赖注入。

  • IoC控制反转 不是一种实现的方法或技巧，而是一种笼统的软件设计思想，它更多表述的是为什么要控制反转、控制反转的目的是什么。
  • DI依赖注入 则是 控制反转 的一种实现方法，怎么去实现 控制反转，依赖注入 只是其中的一种方法。

2.2. 最小自动挡IoC容器 ​

ts
class Container {
  private dependencies: { [key: string]: any } = {}

  register(key: string, dependency: any) {
    this.dependencies[key] = dependency
  }

  getDependency(key: string) {
    if (this.dependencies[key]) {
      return this.dependencies[key]
    }

    throw new Error(`依赖 ${key}未被注册到IoC容器中`)
  }
}

3.4. 装饰器叠加使用 ​

多个装饰器可以叠加使用，它们的执行顺序为从下到上、从小到大（装饰的范围）执行。

多个同类型的装饰器叠加使用，会从最底下的装饰器开始执行，可以看做是 f(g(x)) 的数学式子，同样也是函数组合 compose 的概念。

同类型装饰器执行顺序：

ts
const MessageDecorator: ClassDecorator = (target) => {
  console.log('MessageDecorator')
  target.prototype.success = (data: any) => {
    return {
      code: 200,
      data,
      message: '成功~MessageDecorator'
    }
  }
}

const CallDecorator: ClassDecorator = (target) => {
  console.log('CallDecorator')
  target.prototype.success = (data: any) => {
    return {
      code: 200,
      data,
      message: '成功~CallDecorator'
    }

4. HTTP 协议 ​

HTTP 协议是指客户端和服务端 请求和应答（request、response） 的标准协议，规定传输报文当中，传输和响应按照指定的格式进行的一种规范协议。

所谓的请求和应答，就是客户端发送请求，服务端响应这次的请求，此时就完了一次 HTTP 请求

符合了 HTTP 协议的规范传输报文就是 HTPP 协议。

4.1. HTTP 豹纹 ​

一个完整的 HTPP 协议的报文主要有以下三个部分组成：

  • 起始行（请求、响应行）：描述请求或响应的基本信息
  • 首部字段（请求、响应头）：使用 key-value 的形式更加详细的说明报文
  • 消息正文（请求、响应体）：实际传输的数据，可以是文本、文件、视频、音频等二进制数据，比如前端传输的表单数据，后端响应的 json数据

4.1.1. 请求方法 ​

HTTP/1.1 协议中共定义了八种方法（也叫“动作”）来以不同方式操作指定的资源。

下面是最常见的几种请求方式：

  • GET： 从服务器获取资源
  • POST： 在服务器上创建一个新的资源，常用于表单提交，创建数据
  • PUT： 更新服务器上的资源，返回整个更新后的资源
  • DELETE： 删除服务器上的资源
  • PATCH： 更新服务器上的资源，仅返回更新的部分资源

4.1.2. 常用http响应码 ​

  • 1xx 消息——请求已被服务器接收，继续处理
  • 2xx 成功——请求已成功被服务器接收、理解、并接受
  • 3xx 重定向——需要后续操作才能完成这一请求
  • 4xx 请求错误——请求含有词法错误或者无法被执行，客户端
  • 5xx 服务器错误——服务器在处理某个正确请求时发生错误，500
常见的响应码
200OK从客户端发送的请求，服务端已经正常处理了。
204No Content服务端已经正常处理了,但是响应中没有实体，也不允许有实体。
301Moved Permanently永久性，重定向。表示请求的资源已经拥有了新的 uri，需要重新访问。
302Moved Temporarily临时重定向。
400Bad Request请求报文中存在语法错去。
401Unauthorized请求需要有通过 HTTP 请求的认证信息。
403Forbidden请求被阻止，可能因为某些权限问题，比如访问的文件没有权限等。
404Not Found表示在服务器上没有你要找的资源
500Internal server Error服务器执行程序出现异常

4.1.3. 首部字段 ​

首部字段是构成 http 报文的重要元素，它能起到传递额外重要信息的作用，首部信息一般会提供报文类型、编码和大小、认证信息，缓存策略等信息。

记住常用的几个即可请求：

  • Content-Type：请求体是什么编码格式
  • Content-Length：请求体体积大小
  • Keep-Alive：保持 tcp 时长
  • Accept-Encoding：客户端支持的文件压缩格式
  • Accept：支持返回的文件格式
  • User-Agent：代理客户端的信息，也就是浏览器的信息响应：
  • Content-Type：响应体的数据类型
  • Access-Control-Allow-Origin：是否允许跨域
  • Keep-Alive：保持 tcp 连接的时长

4.2. 数据传输 ​

前后端通过 HTTP 请求通信，前端传递参数给后端，按照位置的不同分为两种：

  • URI 路径参数，参数写在 URI 地址中，一般分为 查询参数 query 、路径参数 url param
  • body 参数，数据参数写在 body 请求体中，有多种数据格式，比如像JSON、form-data、form-urlencoded、xml，一般常见的也就是 JSON和form-data 居多

大多数开发人员是通过请求方式的不同去区分，比如GET不能接收body参数，这种区分显然是不准确的。

因为路径参数 url param 不局限于 GET 请求，其他请求方式也能接收路径参数。并不是不允许这种书写代码的方式，只是不符合规范，我们写的接口要尽量符合 RESTful 规范。

4.3. RESTFul api 接口规范 ​

TIP

全称：Representational State Transfer => REST

直译为：资源表示层状态转移

REST 是 web 服务的一种架构风格，一种设计风格，而不是一种技术手段，写代码，通常规范大于约束。

符合以下这些特点，就能被称为 RESTFul 风格api

  1. 版本号

在请求路径上表明请求资源的版本，能有效解决因为升级系统接口不兼容的问题，通常会在 URL 中保留旧版本，同时兼容新的版本

比如：【GET】/v1/user、【GET】/v2/user

  1. 资源路径

请求路径会被视为服务器的一种资源，在请求路径 URL 中，规定只能包含名词，通过请求方式区分请求的动作，多个单词的情况下，需要使用下划线 _ 划分。

比如：【POST】/v1/user 新增用户

  1. 请求方式

通过请求方式区分此次请求的动机，动作可以划分为四个大类 C R U D，对应 增查改查

  • 【GET】：从服务器上获取资源
  • 【POST】：在服务器创建新的资源
  • 【PUT】：修改服务器上的资源，并返回整个修改后的资源
  • 【DELETE】：删除服务器上的资源
  1. 查询参数

接口应该要接收参数，过滤返回的结果

比如分页：【GET】/v1/api/user?page={num}&size={num}

  1. 状态码

使用合适的状态码，常规的业务错误（表单校验不过）不应该直接抛错返回客户端，应该在接口结果中，返回状态码 200，并且返回描述的错误类型，返回业务状态码。如果非业务错误，可直接响应错误给客户端。

常见非业务错误状态码：

  • 200，正常返回
  • 204，已接收处理，但无实体返回
  • 401，未登录用户
  • 403，未授权的资源
  • 404，找不到资源路径
  • 500，服务器代码异常
  • 503，服务器错误
  1. 响应规范

接口应返回封装统一的 JSON格式的响应体，成功返回正常数据，失败根据是否为业务错误返回状态码、错误信息、错误标识等

成功返回：{code: xxx}

业务错误：{message:'xxxxx',failed:true}

非业务错误：{statusCode:404,message:'Not Found'}

7.2. 作用域 ​

nest 应用，Providers 服务提供者默认是单例模式，这意味着在整个nest 应用中，Providers 只会被实例一次，所有的请求共享相同的 service 服务实例。

这种设计可以更好提高性能，因为减少了 service 服务的实力化次数。

但这种模式，无法实现一些特定的需求，我们可能需要为每个请求提供一个新的 service 服务实例，就需要通过 nest 的一些其他方法来延伸其作用域。

  • DEFAULT：默认作用域，整个应用生命周期只会实例化一次，每次获取的都是初次实例的实例。
  • REQUEST：请求作用域，每次请求都会进行一次实例化，比如获取用户更新的 TOKEN 凭证，我们需要从内存中获取最新的数据返回。
  • TRANSIENt：瞬态作用域，每一个使用 Providers 的模块都会得到一个独立的实例，服务隔离发生在了模块层面。

接下来用同一个例子说明下这三个作用域的区别，添加两个控制器app、user，分别读、写service的属性。

8. 模块 ​

Module 提供了一种模块化方式来组织和管理你的 nest 程序和代码。

被 @Module() 装饰器装饰的类，就具有模块的特征，视为一个模块类，装饰器方法可以传递构建模块的元数据Metadata，比如service服务、controller控制器等。

每个nest应用至少需要一个根模块，根模块是应用组装的入口，每个模块都可以看做是你应用中的子应用，当这些子应用组合起来，就形成一个完整的 nest 应用程序。

默认情况下，模块好比“一个封闭的盒子”，这意味着，如果不在providers元数据中注入服务，是无法使用服务的，并且不导出模块（在exports注入）也无法提供其他模块使用。

因此你可以将把“在模块导出的服务”当做公共接口或者API。

8.1. @Module装饰器 ​

模块就像是一个盒子，这个盒子提供了特定功能的工具，比如providers、controllers、imports导入、exports导出等，在整个盒子的生命周期内，盒子内会共享这些工具。

装饰器接收一个对象参数，类型是ModuleMetadata接口，它具有四个属性字段，皆是数组类型，分别是：

  • imports：导入来自其他模块的providers服务
  • exports：导出providers，其他模块导入时，做到共享的作用
  • providers：模块中所需要的所有service功能类，整个模块共享使用
  • controllers：模块出入口，控制器

nest 的 IoC容器 会根据传入的元数据，准确识别到组件的依赖关系（注入与被注入的关系）。

错误提示找不到依赖，实际是IoC容器 在实例化依赖时，会遍历所有的服务，从它们的依赖注入的属性（依赖注入的参数或使用了@Inject装饰器的参数）进行深度优先遍历，依赖形成了一颗树，它会从叶子节点开始实例化整棵树。遇到循环依赖时，因为循环依赖之间无法形成父子节点关系，遍历就会陷入循环。

理想情况下，程序设计层面不应该出现循环依赖的问题，这是一种错误的设计方式，应该要尽量避免，但是在避无可避的情况下，nest 提供了几种方法来解决这种糟糕的设计。

8.3.1. 最小循环依赖解决方案伪代码 ​

nest 采用的前向引用是将循环依赖的双方都标记上 forwardRef，也当做叶子节点处理，但是不对其进行实例化，而是返回 Object.create() 创建的引用对象。

当遇到了这个标记时，它们调用构造函数后，会用Object.create() 创建的引用对象，去覆盖先前第一步创建的对象，这样先前注入了不完整对象的地方也会变得完整。

ts
class A {
  constructor(b) {
    this.b = b
    console.log(this)
  }
}

class B {
  constructor(a) {
    this.a = a
    console.log(this)
  }
}

function main() {
  const instanceA = Object.create(A.prototype)
  const instanceB = Object.create(B.prototype)

  Object.assign(instanceA, new A(instanceB))

8.4.4. builder 生成动态模块 ​

在 nest 中，一般将创建动态模块的静态方法命名为 register ，但 nest 对此没有硬性规定，比如 nest 中就约定了 3 种方法名，一般方法名不同，代表的含义也不同，所做的事情也不同：

  • register：使用特定的参数配置来创建动态模块，仅供调用模块使用，可以对任意数量模块执行创建动作。

9. 组件 ​

nest 中的组件包含了几种类型，middleware中间件、pipe管道、interceptor拦截器、guards守卫、filter过滤器 等。

它们基本都是基于切面编程（AOP）对请求、方法的增强手段，这些的区别在于作用不同，执行的顺序不同。

  • 中间件：请求进入控制器之前调用的函数，它可以访问请求、响应对象，多个中间件按照定义顺序执行，使用next()方法执行下一个中间件。
  • 守卫：根据运行时存在的某些条件（如权限、角色、TOKEN 等）确定路由是否处理请求，常用于请求的权限校验。
  • 拦截器：有两部分执行时机，一个是在进入控制器方法之前，一个是在离开控制器方法之后（最后的响应拦截）。
  • 管道：数据从管道口流入，进行处理之后从管道尾端流出，经典的用法就是转换数据格式和进行参数校验，在某个方法调用之前，使用管道函数，即可被调用。
  • 过滤器：过滤器是比较特殊的部分，一般是 nest 内置的异常层对程序中所有未处理的异常进行捕获。

9.1.3. 异常过滤 ​

在上一节，我们使用继承自 HttpException 基础类实现自定义异常类，但有时候一些业务操作不一定非要抛出 http 异常，而是抛出业务异常，比如下列格式：

这就要用到 nest 的异常过滤器，异常过滤器必须实现 ExceptionFilter 接口，只有一个 catch() 方法，接收两个参数：

  • 异常对象
  • 上下文对象，当进入到过滤器中，response 对象必须要有内容返回，不然整个程序会造成阻塞，得不到响应而超时

创建一个过滤器：

ts
@Catch()
export class FilterFilter<T> implements ExceptionFilter {
  catch(exception: T, host: ArgumentsHost) {}
}

@Catch() 装饰器的作用是告诉 nest ，当前的类是一个异常过滤器，它接收一个异常类，当没有传递时，会过滤所有的异常。

将过滤器补充完整，返回上面特定的格式：

ts
import {
  ArgumentsHost,
  Catch,
  ExceptionFilter,
  HttpException,
} from '@nestjs/common';
import { Response, Request } from 'express';

@Catch(HttpException)
export class FilterFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    // 取得上下文
    const ctx = host.switchToHttp();
    // 响应对象
    const response = ctx.getResponse<Response>();
    // 请求对象
    const request = ctx.getRequest<Request>();
    // 异常状态码
    const status = exception.getStatus();

    response
      // 响应状态
      .status(status)
      // 响应返回征文
      .send({
        message: exception.message,
        path: request.url,
        method: request.method,
        failed: true,
        data: null,
      });
  }
}

9.4. guards 守卫 ​

一个普通的类，实现了 CanActivate 接口，这样的类就可以被称为守卫（守卫同样也是一个提供者providers）。

守卫的意思是在某个方法前充当《护卫》守护入口，类似于 Vue 的路由鉴权，鉴权正是守卫的最好用例，当用户调用方法，需要校验是否拥有足够的权限 (鉴权、JWT校验、RBAC校验、访问控制) 时，之后链路上的路由方法才可用。

守卫的例子需要搭配 JWT 的生成以及验证。

9.4.1. 基本使用 ​

守卫的基本使用，就是一个普通的类实现了 CanActivate 接口，它会返回一个布尔值

  • 如果返回 true, 将处理用户调用，也就是进入路由方法。
  • 如果返回 false, 则 nest 将忽略当前处理的请求，实际上 nest 会抛出错误 throw new UnauthorizedException() ，也就是会得到403的响应结果，抛出的错误可以结合异常拦截器进行自定义错误。

首先创建一个不实现任何逻辑的守卫：

ts
import { CanActivate, ExecutionContext, Injectable } from '@nestjs/common';
import { Observable } from 'rxjs';

@Injectable()
export class RoleGuard implements CanActivate {
  canActivate(
    context: ExecutionContext,
  ): boolean | Promise<boolean> | Observable<boolean> {
    return true; // nest会放行进入到控制器方法中
    return false; // nest会抛出错误，实际上是触发这行代码 throw new UnauthorizedException(); 得到403的响应结果
  }
}

9.4.2. jwt ​

JWT（Json Web Token） 是一种以JSON作为载体，在不同的服务端之间进行安全的传输信息。

最常见的场景就是授权认证，用户登录过后，服务端发放 jwt token ，后续用户的每个请求都将包含 token ，服务端在处理用户的请求时，会先对 token 进行安全校验，通过之后再进行请求处理。

Token 可以保存，也可以不保存。

保存，是因为要适应大并发场景，比如分布式的服务端，多个服务需要共享 Token 的情况，解析 Token 是需要耗费资源的，若能直接通过快速的判断进行 Token 的有效判断，能节省很多服务器占用，比如 Redis 可以设置过期时间；

不保存，因为用户的信息（除敏感信息之外）都已经存到 Token 里，服务端要做的是对 Token 进行校验，看是否进行加密过且还在有效期内，如果令牌有问题，就拒绝此次请求。

nest 集成了 JWT ，使其变为一个模块，需要安装并从 @nestjs/jwt 中引入，用户数据使用json数据模拟。

9.4.3. passport策略 ​

身份验证具有不同的策略模式，比如登录、注册和正常接口访问是两种不同的策略，登录、注册不需要 Token 令牌，但是正常的接口访问却需要。

在 nodejs 常用 passport 身份验证库作为不同策略的校验，它将身份验证抽象为几个基本步骤，你可以根据实现的策略进行自定义的配置，就能实现不同的策略不同的处理方式。

策略类需要继承来自 @nestjs/passport 包的 PassportStrategy 类，传入对应的策略。

import { PassportStrategy } from '@nestjs/passport';
import { Strategy, IStrategyOptions } from 'passport-local';

import { User } from './dto/user.dto';

export class LocalStragety extends PassportStrategy(Strategy, 'local') {

      passwordField: 'password',

    if (!user) {
      throw new BadRequestException('用户名或密码不正确');
    }

    return user;
  }

        path: request.url,
        method: request.method,
        failed: true,
        data: null,
      });
  }
}

9.4.6. 拓展—RBAC基于角色权限模型 ​

Role-base-assess-control 基于用户的权限访问控制，是一套权限管理模型设计方案，不同的角色拥有不同的权限，用户通过角色与权限进行关联。用户与角色为多对多关系，角色与权限同样为多对多关系。

角色通常指一组具有某些特征的集合，比如：部门/职位/工作职责。

RBAC 模型可以抽象概括为：who是否可以对what进行how的访问操作，并对这个表达式进行判断是否为True 的求解过程。

模型层次 ​

模型定义了四个等级：

  1. 最常用的扁平化模型，用户使用角色来获得权限。

  2. 在0级之上，增加了角色层次结构，比如销售岗的角色分为经理、总监、普通销售，这就是角色的更细粒度的层次拆分。

  3. 在1层之上，增加了职责分离，分为动态和静态的职责分离。

  • 静态职责：
    • 互斥角色：一个用户拥有两个互斥的角色，比如他是警察的同时，又是黑社会，那么他就只能选择其中一个。
    • 基数限制：一个用户拥有的角色是有限的，一个角色拥有的权限同样也是有限的。
    • 先决条件限制：用户想要拥有高级角色，必须先拥有低级角色，比如一个用户相当体制内的领导，那么他必须在基层工作多少年，不允许空降领导。
  • 动态职责：用户拥有多个角色，但是运行时只能激活一个角色，比如sass系统，只能选择一个租户进行操作。
  1. 相当于RBAC3 = RBAC1 + RBAC2，并增加了权限审查。
用户组 ​

在模型之外，还有一个用户组的概念，用户组指的是一系列拥有相同角色的用户集合，在0级的RBAC模型，用户是直接与角色进行关联。

在实际应用中，用户组才是与角色进行绑定，这样就摆脱了用户与角色的直接对应关系，就不需要用户是否存在的先决条件，只要分配到用户组的用户，他就拥有对应权限。

五表设计 ​

经典的RBAC 模型，在数据库设计里面通常有五类表，用户表、角色表、用户角色关联表、权限表、角色权限关联表，这里的权限表指的是多种权限类型的统称，可能还会包含操作权限、功能权限、数据访问控制权限。

9.5. interceptor 拦截器 ​

拦截器是实现 NestInterceptor 接口的类。

拦截器跟守卫干着同样的活，但是拦截器却可以在方法、控制器、响应结果的前后绑定额外的逻辑，比如筛选入参、统一抛出异常、转换参数、统一响应结果等等例子。

拦截器的执行时机是根据 next.handle() 是否被调用来决定的，在调用之前，会在路由方法之前执行；在调用之后，在路由方法之后执行。

学习拦截器需要你对 rxjs 有一定的了解（瞄过文档的程度）。

9.5.1. 切面（AOP） ​

拦截器可以在控制器、路由方法、模块、响应前后添加额外的逻辑，称之为 AOP 切面编程。当我们需要记录与应用的交互时，拦截器就显得很有用，例如日志记录，身份授权和验证，缓存，异常处理，性能监控，请求重试超时处理，响应结果封装等等常用的场景。

创建一个简单的日志记录例子 logging.interceptor.ts

ts
import {
  CallHandler,
  ExecutionContext,
  Injectable,
  Logger,
  NestInterceptor,
} from '@nestjs/common';
import { Observable, tap } from 'rxjs';
import dayjs from 'dayjs';

@Injectable()
export class LoggerInterceptor implements NestInterceptor {
  constructor(private readonly logger: Logger) {}

  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    const start = dayjs();
    return next.handle().pipe(
      tap((response) => {

const CACHE_KEY = Symbol('CACHE_KEY');

@Injectable()
export class CacheInterceptor implements NestInterceptor {
  constructor(private readonly cacheService: CacheService) {}

  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    const isCanCache =
      Reflect.getMetadata(CACHE_KEY, context.getHandler()) ||
      Reflect.getMetadata(CACHE_KEY, context.getClass()); // 判断接口方法是否需要走缓存逻辑

    const request = context.switchToHttp().getRequest();
    if (isCanCache) {
      // 需要走缓存
      const cacheKey = this.redisCacheKey(request.method, request.url);
      const cacheData = this.cacheService.getCacheItem(cacheKey);
      
      console.log(cacheData); // 第二次请求时，如果这条打印了，证明是有缓存的存在

      if (!cacheData) {
        // 无缓存时，把数据设置为缓存
        return next.handle().pipe(
          map((data) => {
            this.cacheService.setCache(cacheKey, data);
            return data;
          }),
        );
      } else {
        // 取出缓存，直接返回
        return of(cacheData);
      }
    }
    return next.handle();
  }

  private redisCacheKey(method: string, url: string): string {
    return `${method}:${url}`;
  }
}

9.6. pipe 管道 ​

管道（Pipes）是一种用于处理输入数据的机制。管道允许你对进入应用程序的数据进行转换、验证和转变。它就像一根水管，数据从开端（客户端）处流入，经过管道处理之后，从管道的末端流出供给 nest 程序使用。

它们是NestJS中的一种重要概念，用于帮助保证输入数据的有效性和一致性。

管道有两个典型的应用场景：

  • 转换：管道将输入数据转换为你所需要的数据输出，比如数字转字符。
  • 验证：对输入的数据进行验证，如果验证成功继续传递，验证失败则会抛出异常。

这些数据是由控制器的路由方法进行处理，nest 在调用这些路由方法之前插入管道，管道会拦截方法的调用，然后进行处理后，将处理后的数据调用原方法。

实际上就是通过参数装饰器，在路由方法调用之前，对管道使用 Reflect.metadata 储存起来，然后在数据传输（如：@Query）时进行处理。

9.6.1. 内置管道 ​

nest 内置了几个开箱即用的管道，分别为：

  • ValidationPipe：结构验证，有类结构和对象结构的验证，可对 DTO 进行属性限制验证。
  • ParseIntPipe：将数据转换为 Number 类型。
  • ParseFloatPipe：将数据转换为 Float 浮点类型。
  • ParseBoolPipe：将数据转换为 Boolean 布尔类型，符合 ES 的隐式转换。
  • ParseArrayPipe：将数据转换为 Array 数组类型。
  • ParseUUIDPipe：将数据转换为 UUID 类型。
  • ParseEnumPipe：将数据转换为 Enum 枚举类型。
  • DefaultValuePipe：提供默认值，当接收到null 或 undefined 时，程序可能会抛出异常，为了程序正常运行，可以给参数提供默认值。

如果转换失败，则会抛出类型不符的异常。

9.6.2. 基本使用 ​

管道可以是参数作用域、方法作用域、控制器作用域或是全局作用域（如果要用到依赖注入时，要在 app.module 中使用自定义依赖注入），与参数作用域进行搭配时，直接作为参数传递即可，其他作用域则需要 @UsePipe 装饰器。

    //    }

    // 1. 根据DTO类,借助 class-transformer 插件实例化对象
    const userDto = plainToInstance(metadata.metatype, value);

    // 2. 根据DTO类属性的 装饰器验证规则rules，获得一组特殊的Dto对象，它们包含验证rules，然后对这组数据进行校验
    const errors = await validate(userDto);

    // 3. 如果errors长度为0，说明所有验证通过，否则校验失败，返回错误信息给前端
    if (errors.length) {
      throw new HttpException(
        errors.map((item) => Object.values(item.constraints)[0]),
        HttpStatus.BAD_REQUEST,
      );
    }

    return value;
  }
}

9.7. 各个组件请求链路 ​

客户端发送一个请求，nest 各个组件的生命周期，即为请求链路

先说结论：（如有多个同样的组件，就按注册顺序执行）

  1. 中间件：全局 → 模块
  2. 守卫：全局 → 控制器 → 路由
  3. 拦截器，执行handle之前：全局 → 控制器 → 路由
  4. 管道：全局 → 控制器 → 路由 → 参数
  5. 控制器内部
  6. 拦截器，执行handle之后： 路由 → 控制器 → 全局
  7. 异常过滤，特殊层穿插在每个生命周期，只要有异常都会进入异常层：路由 → 控制器 → 全局
